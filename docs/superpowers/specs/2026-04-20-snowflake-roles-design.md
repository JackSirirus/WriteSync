# WriteSync 共笔 — 雪花写作法角色拆解设计

> 日期：2026-04-20
> 状态：初版

---

## 1. 设计目标

将雪花写作法（Snowflake Method）按角色拆解，设计适合网文创作场景的 Multi-Agent 协作架构。

---

## 2. 角色定义

### 2.1 角色总览（15个）

| 序号 | 角色 | 职责 |
|------|------|------|
| 1 | **选题 Agent** | 根据平台特性提出选题建议（题材/风格/卖点） |
| 2 | **选题检查 Agent** | 检查选题是否合理，可引用平台特性 |
| 3 | **策划 Agent** | 基于选题做一句话摘要 + 三句扩展 + 一段式大纲 |
| 4 | **策划检查 Agent** | 检查策划是否偏离用户原意，可引用用户原始需求 |
| 5 | **角色 Agent** | 人物设定、关系网络、成长弧线 |
| 6 | **角色检查 Agent** | 检查角色卡质量，可引用策划输出 |
| 7 | **世界观 Agent** | 设定体系、力量系统、地理社会结构，自检 |
| 8 | **章纲 Agent** | 章节规划、情节点安排、伏笔植入 |
| 9 | **章纲检查 Agent** | 检查章纲一致性，可引用策划+角色 |
| 10 | **文笔 Agent** | 按章写初稿（3000-5000字/章） |
| 11 | **文笔检查 Agent** | 检查该章初稿，可引用章纲+角色 |
| 12 | **编辑 Agent** | 质量检查、逻辑连贯性，该章内自检 |
| 13 | **节奏 Agent** | 情绪节奏、爽点铺设、章节结尾钩子设计，该章内自检 |
| 14 | **校对 Agent** | 错别字、标点、格式检查，该章内自检 |
| 15 | **用户交互 Agent** | 跨阶段与用户确认，协调各 Agent 与用户的沟通 |

### 2.2 角色详细说明

#### 选题 Agent
- **输入**：用户模糊想法 + 目标平台
- **输出**：选题建议列表（题材/类型/核心卖点/竞品对比）
- **输出格式**：结构化选题卡，包含平台适配分析

#### 选题检查 Agent
- **输入**：选题 Agent 输出 + 平台特性数据
- **检查点**：选题是否符合平台调性、是否有差异化竞争力、题材是否过饱和
- **可引用**：平台历史爆款数据、用户画像

#### 策划 Agent
- **输入**：已确认的选题
- **输出**：一句话摘要 + 三句话扩展 + 一段式大纲

#### 策划检查 Agent
- **输入**：策划 Agent 产出 + 用户原始需求
- **检查点**：是否偏离用户意图、逻辑是否自洽
- **可引用**：用户原始需求文档

#### 角色 Agent
- **输入**：策划阶段产出
- **输出**：角色卡（性格/目标/关系/成长弧线/场景描写要点）

#### 角色检查 Agent
- **输入**：角色卡 + 策划输出
- **检查点**：角色是否与故事主题契合、性格是否可区分、关系网是否清晰
- **可引用**：策划输出（确保角色符合故事基调）

#### 世界观 Agent
- **输入**：策划输出 + 角色设定
- **输出**：世界观设定文档（力量体系/地理/社会结构/历史）
- **自检**：内部检查一致性

#### 章纲 Agent
- **输入**：策划 + 角色 + 世界观
- **输出**：章纲（每章核心事件/人物状态变化/伏笔/情绪节奏/预计字数）

#### 章纲检查 Agent
- **输入**：章纲 + 策划输出 + 角色设定
- **检查点**：章纲是否承接策划意图、人物进展是否合理、伏笔是否有铺垫
- **可引用**：策划输出（故事主线）+ 角色设定（人物弧线）

#### 文笔 Agent
- **输入**：该章章纲 + 角色设定 + 世界观
- **输出**：该章初稿（3000-5000字）
- **注意**：按章节粒度工作，每章独立产出

#### 文笔检查 Agent
- **输入**：该章初稿 + 该章章纲 + 角色设定
- **检查点**：是否按章纲展开、文笔风格是否一致、人物台词是否贴合角色
- **可引用**：该章章纲 + 角色设定

#### 编辑 Agent
- **输入**：该章初稿
- **输出**：修订意见（自查清单：设定层/人物层/结构层/文字层）
- **自检**：内部执行质量检查

#### 节奏 Agent
- **输入**：该章修订稿 + 该章章纲
- **输出**：节奏优化建议（爽点位置/高潮节奏/章节断点/钩子设计）
- **自检**：内部检查节奏感

#### 校对 Agent
- **输入**：该章节奏优化稿
- **输出**：终版文本（错别字/标点/格式）
- **自检**：内部完成校对

#### 用户交互 Agent
- **职责**：跨阶段与用户确认，负责协调各 Agent 与用户的沟通
- **介入时机**：选题后、策划后、角色后、世界观后、章纲后、每章初稿后

## 3. 核心工作流程

### 3.1 阶段划分

```
阶段1：选题
用户想法 → 选题 Agent → 选题检查 Agent → 用户交互 Agent【确认选题+平台】

阶段2：策划
选题输出 → 策划 Agent → 策划检查 Agent → 用户交互 Agent【确认方向】

阶段3：角色
策划输出 → 角色 Agent → 角色检查 Agent → 用户交互 Agent【确认人设】

阶段4：世界观
角色输出 → 世界观 Agent → 自检 → 用户交互 Agent【确认设定】

阶段5：章纲
策划+角色+世界观 → 章纲 Agent → 章纲检查 Agent → 用户交互 Agent【确认章纲】

阶段6：每章循环（核心差异化点）
章纲 → 文笔 Agent → 文笔检查 Agent → 用户交互 Agent【确认该章】
→ 编辑 Agent → 节奏 Agent → 校对 Agent → 发布 → 下一章循环

阶段7：全稿完成
所有章节发布 → 汇总输出
```

### 3.2 网文特殊性：章节粒度驱动

网文与传统出版的关键差异在于**每天更新的章节驱动模式**，因此：

1. **修订不是全稿完成后统一做**，而是每写完一章就进入编辑+节奏+校对流程
2. **校对 Agent 也在每章后执行**，不等全稿
3. **用户交互 Agent 的确认粒度是章节级**，不是整个大纲或整份初稿
4. **节奏 Agent 负责章节结尾钩子设计**，确保每章留有悬念吸引读者继续阅读

---

## 4. 检查 Agent 设计原则

1. **每个主 Agent 配备独立检查 Agent**（世界观/编辑/节奏/校对自检）
2. **关键步骤（策划/角色/章纲/文笔）配检查 Agent**，其他步骤自检
3. **检查 Agent 可往前引用**，确保当前步骤和之前步骤的一致性
4. **用户交互 Agent 是共享的**，跨阶段负责与用户确认

## 7. 平台数据来源方案

### 7.1 混合方案（已确定）

- **人工维护核心知识库**：每平台一个 Markdown 文件，存储各平台特性
- **定期爬取榜单补充**：各平台排行榜数据作为参考数据

### 7.2 平台知识库维度（已确定）

每平台 Markdown 文件包含以下维度：

| 维度 | 说明 |
|------|------|
| 题材偏好 | 目前流行什么题材，哪些题材已过饱和 |
| 风格倾向 | 文风特点（小白文/老白文/轻小说风） |
| 更新节奏 | 日更/隔日更要求，字数要求 |
| 读者画像 | 主要读者群体，年龄/性别/喜好 |
| 爆款规律 | 什么元素容易爆（战神/穿越/系统等） |
| 禁忌/审核 | 什么内容不能写，什么元素有风险 |
| 变现模式 | 订阅/打赏/免费/付费模式 |

### 7.3 目录结构

```
docs/platforms/
├── 起点.md
├── 番茄.md
├── 飞卢.md
├── 纵横.md
└── ...
```

## 9. 技术栈选型

### 9.1 确定技术栈

- **语言**：Python
- **框架**：LangGraph（推荐）

### 9.2 LangGraph 选型理由

| 需求 | LangGraph 支持情况 |
|------|-------------------|
| 多 Agent 协作 | ✅ 支持 |
| Checkpoint/状态持久化 | ✅ 内置，每章可保存状态 |
| 人机交互（每步确认） | ✅ 原生 human-in-the-loop |
| 往前引用（跨节点状态） | ✅ State 共享，检查 Agent 可读之前产出 |
| 章节粒度工作流 | ✅ 循环 + Checkpoint 天然支持 |
| 图结构可视化 | ✅ 清晰描述 15 个 Agent 的工作流 |

### 9.3 技术架构预览

```
StateGraph
├── 节点：15 个 Agent 对应 15 个 Node
├── 边：固定顺序 + 条件跳转
├── Checkpoint：每章完成后保存状态
└── 人机交互：interrupt + resume 模式
```
