# WriteSync v0.4.0 — 分卷雪花：网文自适应写作架构设计

> 日期：2026-06-10
> 状态：已实现 (2026-06-12)

## 1. 背景与动机

### 1.1 问题

WriteSync v0.3.0 基于雪花写作法（Snowflake Method）搭建了主 Agent + 子 Agent 编排架构。实践证明雪花写作法非常适合 AI 辅助写作——结构化扩展、逐层验证、每步可讨论修改。但它有一个根本问题：

**雪花写作法为传统单本小说设计，未考虑网文的连载特性**。

网文的核心差异：
- **连载模式**：不是一次写完，而是逐卷发布，读者反馈实时影响后续
- **钩子驱动**：每章末尾必须有钩子，读者才会追更
- **爽点节奏**：需要精确控制爽点的密度、强度和类型分布
- **平台分化**：起点/飞卢/番茄/纵横的读者预期和算法机制完全不同
- **开篇即决战**：黄金三章决定生死，前三章留不住人后面全是白费

### 1.2 目标

在保持雪花写作法"结构化 AI 辅助"核心优势的前提下，融入网文必需机制，做到：

1. **开篇抓力**：黄金三章专项强化，强制钩子/爽点/冲突密度
2. **全平台通吃**：一套引擎，按平台参数自适应
3. **钩子矩阵 + 爽点曲线**：让 AI 不只是"写出来"，而是"写得有网文味"
4. **辅助而非替代**：所有校验只做提醒，人始终是最终决策者
5. **不增加 LLM 调用**：新增模块尽量走纯规则/策略计算

## 2. 整体架构：两层结构

```
┌────────────────────────────────────────────┐
│ 第一层：全书规划（项目初始化，一次性）        │
│                                             │
│ ① 选题 → ② 一句话核心 → ③ 全书大纲（粗树干） │
│ ④ 角色设计（Snowflake 方式，含弧线+反差）    │
│ ⑤ 世界观/力量体系                           │
│ ⑥ 平台策略绑定                              │
└──────────────────┬─────────────────────────┘
                   │
     ┌─────────────┴─────────────┐
     ▼                           ▼
┌──────────┐               ┌──────────┐
│ 第一卷   │               │ 第 N 卷   │
│ 微循环   │               │ 微循环   │
│          │               │          │
│ ① 卷纲   │（来自全书大纲  │ ① 卷纲   │
│ ② 钩子矩阵│  对应段落展开) │ ② 钩子矩阵│
│ ③ 爽点曲线│              │ ③ 爽点曲线│
│ ④ 章纲   │               │ ④ 章纲   │
│ ⑤ 逐章写作│              │ ⑤ 逐章写作│
│ ⑥ 卷审查 │               │ ⑥ 卷审查 │
│          │               │          │
│ 角色变更 │──→ 更新角色 ──→│ 角色变更 │
│ 检测     │               │ 检测     │
└──────────┘               └──────────┘
     │                           │
     └──────────┬────────────────┘
                ▼
         ┌──────────┐
         │ 全书审查  │
         └──────────┘
```

**层级关系**：
- 全书大纲是"粗树干"，卷纲是其对应段落的展开细化，不是另起炉灶
- 角色设计一次完成，每卷末仅检测变更（新登场/弧线推进/关系变化/退场），不全量重做
- 钩子矩阵和爽点曲线在每卷微循环内独立规划

## 3. 核心新增模块

### 3.1 钩子矩阵（Hook Matrix）

将现有 `docs/techniques/钩子技法.md` 从静态参考文档升级为动态执行引擎。

每卷章纲生成时，主 Agent 产出一张钩子规划表：

```
  章序  钩子类型    强度  钩子内容（brief）         衔接章
  ────  ─────────  ────  ──────────────────────    ────
  Ch1   悬念钩子    ★★★★  神秘黑石中传出心跳声…       Ch2开头
  Ch2   冲突钩子    ★★★   家族长老当众宣布废黜主角      Ch3开头
  Ch3   期待钩子    ★★★★  地宫入口浮现，传承即将开启     Ch4开头
  Ch4   危机钩子    ★★★   守护兽苏醒，入口开始坍塌       Ch5开头
  Ch5   反转钩子    ★★★★★ 传承竟是陷阱，主角反杀夺舍者    Ch6开头
```

**钩子类型**（来自 `钩子技法.md`）：悬念 / 冲突 / 期待 / 危机 / 反转 / 情感

**强度等级**：★ ~ ★★★★★

**约束规则**（纯规则校验，不调 LLM）：
- 同一钩子类型不连续出现超过 2 次
- 每 5 章至少 1 个 ★★★★ 以上强度的钩子
- 开篇卷（Ch1-3）：全部 ★★★★ 以上
- 类型分布建议：悬念 30% / 冲突 25% / 期待 20% / 危机 15% / 反转 7% / 情感 3%
- 卷末章强制 >= ★★★★

**Writer 指令**：每章写作时，钩子卡片注入 prompt，强制"章末最后一句话必须是钩子落地句"。

### 3.2 爽点曲线（Pleasure Point Curve）

每卷生成一条爽点密度曲线，Writer 按规划分配爽点：

```
爽点强度
  ★★★★★ │          ★  (Ch5 小高潮)
  ★★★★  │    ★           ★  (Ch3)      ★ (Ch8 中高潮)
  ★★★   │  ★    ★    ★     ★    ★    ★
  ★★    │ ★               ★          ★
  ★     │★                                  ★
        └──┬──┬──┬──┬──┬──┬──┬──┬──┬──→ 章节
          Ch1 Ch2 Ch3 Ch4 Ch5 Ch6 Ch7 Ch8 Ch9 Ch10

  类型分布目标：打脸30%｜突破25%｜收获20%｜反转15%｜智胜10%
```

**爽点类型**（来自 `爽点设计.md`）：打脸 / 突破升级 / 收获获得 / 复仇 / 逆袭反转 / 智胜碾压 / 情感满足 / 身份揭晓 / 伏笔回收

**密度规则**：
- 每章至少 1 个微爽点（★）
- 每 3-5 章 1 个小爽点（★★★）
- 每 8-10 章 1 个中爽点（★★★★）
- 卷末章自动设为爆点（★★★★★）
- 同一爽点类型不连续使用
- 开篇 Ch1 必须包含爽点（哪怕微爽点），遵循"开篇不压主"

### 3.3 平台策略层（Platform Strategy Layer）

项目初始化时用户选定平台，全流程自动注入对应参数：

| 参数 | 起点 | 飞卢 | 番茄 | 纵横 |
|------|------|------|------|------|
| 爽点密度目标 | 中高 | 极高 | 高 | 中 |
| 钩子强度下限 | ★★★ | ★★★★ | ★★★★ | ★★★ |
| 开篇压主容忍 | 低 | 零容忍 | 零容忍 | 低 |
| 文风要求 | 中性 | 轻松/梗密集 | 轻松/快节奏 | 文学性 |
| 日更参考 | 3000-5000 | 5000-10000 | 4000-6000 | 4000-6000 |
| 系统面板偏好 | 可选 | 强烈推荐 | 强烈推荐 | 不推荐 |
| 审核严格度 | 最严 | 中等 | 较松 | 中等 |
| 黄金三章强化 | 标准 | 极度强化 | 极度强化 | 标准 |

**注入方式**：以系统指令片段注入每个 Agent prompt 的 `## 当前平台约束` 段。不额外调 LLM。

## 4. 黄金三章专项模式

第一卷 Ch1-3 自动触发，施加硬约束。Ch4 起自动解除。

```
第一卷 Ch1-3 强制参数覆盖：
  ┌──────────┬──────────────────────────────────────┐
  │  约束项   │  硬性要求                              │
  ├──────────┼──────────────────────────────────────┤
  │ 开场方式  │  30字内进入冲突/悬念，禁止环境描写开篇    │
  │ MC 出场   │  200字内完成主角身份+处境+特殊性展示     │
  │ 金手指    │  Ch1 必须暗示或展现金手指的存在           │
  │ 爽点      │  Ch1 必需 1 个微爽点，Ch2 必需 1 个小爽点 │
  │ 钩子      │  每章末 ★★★★ 以上，Ch3 末 ★★★★★        │
  │ 压抑容忍  │  主角被压制时长 ≤ 全章 15%               │
  │ 对话占比  │  ≥ 50%，用对话推进而非叙述               │
  │ 毒点检测  │  自动扫描：送女/绿帽/跪舔/圣母/无故被辱   │
  └──────────┴──────────────────────────────────────┘
```

**章节模板**：

```
Ch1 结构：
  ① 钩子开场（30字，冲突/悬念/反常）
  → ② MC 处境展示（150字）→ ③ 金手指暗示/展现
  → ④ 微小收获/反击（微爽点）→ ⑤ 章末强钩子（★★★★）

Ch2 结构：
  ① 承接上章钩子 → ② 冲突升级（对手施压/环境恶化）
  → ③ MC 尝试应对 → ④ 小爽点（第一次明显反击/收获）
  → ⑤ 更强钩子（★★★★★），暗示更大危机或机遇

Ch3 结构：
  ① 快速承接 → ② 冲突推向小高潮
  → ③ MC 关键抉择/突破 → ④ 第一个中爽点（爆发）
  → ⑤ 章末钩子转向：揭示更大的世界观/阴谋/目标，锁定追读
```

## 5. 卷内微循环

每卷执行 6 步，卷纲来源于全书大纲对应段落：

```
┌──────────────────────────────────────────────┐
│ ① 卷纲策划                                   │
│   输入：全书大纲对应卷段落 + 上卷末尾状态       │
│   产出：卷级一句话核心 + 冲突主线               │
│   不重做角色/世界观                            │
├──────────────────────────────────────────────┤
│ ② 钩子矩阵设计（主 Agent 决策层，不调子 Agent） │
│   输入：卷纲 + 章数                           │
│   产出：按章分配钩子类型 / 强度 / 内容 / 衔接章  │
│   校验：类型多样性 ✓ 强度下限 ✓ 连续重复 ✓      │
├──────────────────────────────────────────────┤
│ ③ 爽点曲线规划（主 Agent 决策层，不调子 Agent） │
│   输入：卷纲 + 钩子矩阵 + 卷内高潮点           │
│   产出：按章分配爽点类型 / 强度 / 字数占比建议    │
│   校验：密度达标 ✓ 类型不连用 ✓ 卷末爆点 ✓     │
├──────────────────────────────────────────────┤
│ ④ 章纲生成（Outline Agent）                  │
│   输入：卷纲 + 钩子矩阵 + 爽点曲线 + 角色状态    │
│   每章：标题 / 核心事件 / POV / 字数 /         │
│          钩子卡片 / 爽点场景 / 节奏标记          │
│   注入：黄金三章技法（若 Ch1-3）               │
├──────────────────────────────────────────────┤
│ ⑤ 逐章写作（Writer → 用户确认 → Proofreader） │
│   详见 §6                                     │
├──────────────────────────────────────────────┤
│ ⑥ 卷末审查（Novel Review Agent）             │
│   检查：卷级弧线完整性 / 爽点实际密度 /         │
│          钩子落地率 / 字数分布 / 伏笔收放        │
│   产出：卷审查报告 + 下卷策略建议               │
│   角色变更检测 → 更新角色状态（不重做）          │
└──────────────────────────────────────────────┘
```

## 6. 逐章写作层

定位：**辅助写作，而非自动写作**。所有校验以提醒形式展示，用户始终是决策者。

```
┌──────────────────────────────────────────────┐
│ Writer Agent                                  │
│                                               │
│ 输入组装（五合一）：                            │
│ ① 本章章纲（标题 + 核心事件 + 字数）           │
│ ② 钩子卡片（类型 / 强度 / 落地句 / 衔接章）    │
│ ③ 爽点场景（类型 / 强度 / 建议字数占比）       │
│ ④ 写作上下文（角色快照 + 前章回顾 + 伏笔状态） │
│ ⑤ 平台风格指令                                │
│                                               │
│ + 黄金三章强制模板（仅 Ch1-3 自动叠加）        │
│                                               │
│ 输出：3000-5000 字正文                         │
│                                               │
│ → 呈现给用户                                   │
│                                                │
│ 用户操作：接受 / 自己修改 / 提意见让 AI 重写    │
├──────────────────────────────────────────────┤
│ 辅助检查清单（同步展示，不阻断）                │
│                                               │
│ 纯规则运行，结果列给用户：                      │
│                                               │
│ ✓ 钩子落地：末段检测到悬念钩子信号             │
│ ⚠ 爽点密度：本章爽点占比 8%（预设 12%）        │
│ ⚠ 毒点风险：第 243 行"跪地求饶"，请确认合理性  │
│ ✓ 字数范围：3847 字                           │
│ ✓ 黄金三章对话占比：52%（仅 Ch1-3 检查）       │
│                                               │
│ 用户决定：忽略 / 手动改 / 让 AI 修             │
├──────────────────────────────────────────────┤
│ Proofreader Agent（用户确认正文后触发）         │
│                                               │
│ 校对 + 节奏复核：                              │
│ · 错别字 / 语法 / 标点                         │
│ · 钩子强度复核（是否达到预设 ★ 级别）          │
│ · 爽点密度复核                                │
│                                               │
│ 输出：修正后正文 + 校对报告                     │
│ → 再次呈现给用户终审                            │
├──────────────────────────────────────────────┤
│ 上下文更新（终稿确认时触发）                    │
│ 详见 §7                                       │
└──────────────────────────────────────────────┘
```

### 校验指标说明

| 校验项 | 方法 | 输出 |
|--------|------|------|
| 钩子落地 | 末段关键词匹配（悬念词/冲突词/危机词/反转词） | ✓/⚠ |
| 爽点密度 | 正则匹配爽点场景模板，算字数占比 | ⚠ 偏差提醒 |
| 毒点扫描 | 关键词黑名单（送女/绿帽/跪舔/圣母/无故被辱） | ⚠ 位置标记 |
| 字数范围 | 统计字数 | ✓/⚠ |
| 黄金三章 | Ch1-3 额外检查：开场方式/对话占比/压主时长 | ⚠ 仅提醒 |

## 7. 上下文引擎改造

现有 6 子系统扩展为 8 子系统，融入钩子/爽点/平台维度的感知：

```
每章终稿确认后触发更新：

  旧 6 子系统          →  新 8 子系统
  ─────────────────────────────────────
  ① 角色变化提取       →  ① 角色变化提取（不变）
  ② 一致性检测         →  ② 一致性检测（不变）
  ③ 前章摘要           →  ③ 前章摘要（不变）
  ④ 伏笔追踪           →  ④ 伏笔追踪 + 钩子关联  ★ 新增
```

- 伏笔和钩子打通：钩子落地 = 伏笔推进信号，伏笔搁置太久 = 提醒设钩子回收

- 平台偏差预警：检测到当前风格偏离平台预期时提醒
- 所有 ⚠ 只做提醒，不强制阻断

## 8. 主 Agent 编排改造

现有 7 子 Agent 不变。改造的是主 Agent 决策逻辑——按阶段自动切换三种模式：

### 8.1 三种决策模式

| 模式 | 触发条件 | 决策链 |
|------|---------|--------|
| **规划模式** | 项目初始化 / 进入新卷 | story → character → world → outline（一鼓作气连续调度） |
| **编排模式** | 卷内逐章 | writer → 等待确认 → proofreader → 等待确认（循环到卷末） |
| **审查模式** | 卷末 / 全书末 | novel_review → 等用户决策（继续 / 修改 / 完结） |

### 8.2 决策上下文

每次 thinking 时注入：

```
当前阶段：第一卷 / 微循环② 钩子矩阵 / Ch5 写作
全书进度：V1-Ch4 已完稿，共 5 卷 100 章计划
平台模式：番茄（爽点高密 / 开篇零压主 / 系统面板偏好）
黄金三章：已过（Ch1-3 完成，强制约束已解除）
上次用户反馈："这章结尾不够燃，钩子太软"
上下文快照：主角刚突破元婴，伏笔#2 已收回...
```

### 8.3 不新增子 Agent 的决策

钩子矩阵和爽点曲线由主 Agent 在规划模式中直接生成，写入章纲。不额外调子 Agent，不增加 API 调用。

### 8.4 Assistant 定位贯穿

```
主 Agent decision: call_agent → writer
   ↓
Writer 输出正文
   ↓
辅助检查清单（纯规则，≥80ms 完成）
   ↓
SSE 事件推给前端：正文 + 检查结果
   ↓
用户看到：
  ┌─ 正文
  ├─ ✓ 钩子落地 ✓
  ├─ ⚠ 爽点密度偏低
  ├─ ✓ 字数范围
  └─ [接受] [自己修改] [让 AI 重写]

人始终在方向盘上。
```

## 11. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 钩子矩阵/爽点曲线主 Agent 生成质量不稳定 | 加纯规则校验层（§3 约束规则）；钩子/爽点卡片用单独 `complete()` 生成（不合并到 `OrchestratorDecision` 路由），返回 markdown 表格→正则解析 |
| 钩子矩阵无限重生成 | 最多 3 次：3 次仍不通过→降级为"全部悬念钩子 ★★★"，标记 `auto_degraded=True`，用户可见并决定继续或手动调整 |
| 主 Agent prompt 膨胀 | 决策上下文分 L1/L2，L1 只传阶段+进度（≤200 字），L2 按模式按需注入 |
| 黄金三章模板限制创作自由度 | 仅 Ch1-3 生效，Ch4 自动解除；所有约束是"建议+提醒"，用户可忽略 |
| 平台参数一刀切 | 参数可按卷调整，卷间可修改平台策略 |
| 上下文引擎新增子系统增加复杂度 | 钩子/爽点统计走纯规则（计数+正则），不调 LLM |
| Writer 输入 5 合 1 prompt 过长 | 钩子卡片 ≤100 字，爽点场景 ≤80 字，上下文 ≤800 字，总量在原有水平 |
| 旧项目默认参数不匹配 | 迁移时根据 `story.platform` 字段推断；无法推断用最保守的起点参数 |
| 校验精度不足（关键词/正则误判） | 所有校验标注"建议"，用户可忽略，误判不阻断流程 |
| 主 Agent HookCard JSON 解析失败 | markdown 表格→正则解析 + JSON fallback，两种解析器并行，任一成功即通过 |

## 13. 数据模型：卷级抽象

### 13.1 卷状态（VolumeState）

在 `state_types.py` 中新增（不破坏现有字段）：

```
WriteSyncState 新增字段：
  volumes: list[VolumeState]         # 全书卷列表
  current_volume_index: int          # 当前卷索引（0-based）
  platform_profile: PlatformProfile  # 当前平台策略（可卷间修改）
```

`VolumeState` 定义：

```
@dataclass
class VolumeState:
    index: int                    # 卷序号（1-based）
    title: str                    # 卷标题
    one_sentence: str             # 卷一句话核心
    main_conflict: str            # 卷级冲突主线
    chapter_indices: list[int]    # 本章节在全书中的索引范围（0-based）
    hook_matrix: list[HookCard]          # 钩子矩阵（按章排列）
    pleasure_curve: list[PleasurePointCard]  # 爽点曲线（按章排列）
    outline_confirmed_at: str     # 卷纲确认时间
    status: str                   # "planning" | "writing" | "reviewing" | "completed"
```

关键规则：
- `chapter_indices` 是全书章索引范围（如 `[0, 1, 2, ..., 19]`），章数据仍然存在于 `drafts` 中（`drafts[0]` 到 `drafts[19]`），卷只记录哪些章属于它
- 章纲存储在 `chapter_outlines[chapter_index]`，钩子/爽点来自卷的 `hook_matrix` 和 `pleasure_curve`
- 每章从对应卷的 `hook_matrix[chapter_index_in_volume]` 读取钩子卡片

### 13.2 新数据类型

```
@dataclass
class HookCard:
    chapter_index: int            # 本章在卷内的索引
    hook_type: str                # "悬念" | "冲突" | "期待" | "危机" | "反转" | "情感"
    strength: int                 # 1-5（→ ★ 颗数）
    content: str                  # 钩子内容简述（≤50 字）
    connect_chapter: int          # 衔接的下一章索引

@dataclass
class PleasurePointCard:
    chapter_index: int
    pp_type: str                  # "打脸" | "突破升级" | "收获获得" | "复仇" |
                                  # "逆袭反转" | "智胜碾压" | "情感满足" | "身份揭晓" | "伏笔回收"
    strength: int                 # 1-5
    description: str              # 爽点场景简述（≤50 字）
    word_ratio_target: float      # 建议字数占比（如 0.12 = 12%）

@dataclass
class PlatformProfile:
    platform: str                 # "起点" | "飞卢" | "番茄" | "纵横"
    pleasure_density: str         # "极高" | "高" | "中高" | "中"
    hook_strength_min: int        # 钩子强度下限（1-5）
    suppress_tolerance: str       # "零容忍" | "低"
    style_requirement: str        # "轻松/梗密集" | "轻松/快节奏" | "中性" | "文学性"
    daily_ref_words: tuple[int, int]    # 日更参考范围
    system_panel_preference: str  # "强烈推荐" | "可选" | "不推荐"
    audit_strictness: str         # "最严" | "中等" | "较松"
    golden_three_boost: str       # "极度强化" | "标准"
```

### 13.3 与现有 ChapterOutline 的关系

现有 `ChapterOutline.hook_at_end: str = ""` **保留不变**（V1 兼容），但 V2 流程中改用卷的 `HookCard` 驱动。两者关系：
- V1 图流程：使用 `ChapterOutline.hook_at_end`（文本描述）
- V2 编排器流程：使用 `VolumeState.hook_matrix[i].content`（结构化钩子），Writer prompt 从钩子卡片组装
- 两者不互相同步——它们属于不同流程

## 15. 边界情况处理

### 15.1 平台参数卷间切换

用户可以在卷审查完成后修改平台策略，切换到下一卷时生效：
- 已完成的卷不重新评估（历史数据不变）
- 新卷的钩子矩阵/爽点曲线按新平台参数生成
- 上下文引擎从新卷开始切换平台偏差检测基准

### 15.2 卷内章数变化

用户可能在写作中途增删章：
- **增加章**：新章继承卷默认参数（钩子=悬念★★★，爽点=微爽点★），上下文统计自动扩展
- **删除章**：钩子矩阵/爽点曲线重新索引，删除章对应的卡片移除
- 累积影响超过 3 章变化时，前端提示"建议重新规划本卷钩子和爽点"

### 15.3 卷内情节重大变更

用户反馈导致大纲大幅变化时：
- 若影响面 < 3 章：钩子/爽点卡片手工调整（前端直接编辑 HookCard/PleasurePointCard）
- 若影响面 ≥ 3 章：触发"部分重规划"→ 主 Agent 重新生成受影响章的钩子/爽点卡片
- 保留未受影响章的已有卡片不变

### 15.4 约束优先级

当多个约束冲突时，取最高值：

```
黄金三章（★★★★） > 平台快节奏/零压主 > 平台常规钩子下限
```

例：纵横平台钩子下限 = ★★★，但 Ch1-3 强制 ★★★★ → 取 ★★★★。
